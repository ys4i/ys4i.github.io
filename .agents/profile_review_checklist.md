# コーディングエージェント向け：/profile 校正・添削ガイド（Rubric＋チェックリスト）

作成日: 2025-12-30（JST）  
対象: `https://ys4i.github.io/profile/` の **/profile** ページ（採用担当者が最初に見る入口）  
目的: **人事（非エンジニア）でも数秒で理解でき、興味に応じてエンジニア向け詳細へ遷移できる**ように、ページ分量・構成・文言・見せ方を校正する。

---

## 0. 背景（このページの役割と前提）

### 想定読者と閲覧状況
- 一次読者: **人事（非エンジニア）**。初見でスキャンして「採用検討に値するか」を高速判断する。
- 二次読者: **現場エンジニア**。興味を持った後に、Projects/個別ページで技術・再現性・設計意図を確認する。
- 表現方針: **ビジュアル中心**（ただし、画像を増やしすぎて要点が埋もれないこと）。

### /profile の役割（サイト全体の入口）
- /profile は「全部説明するページ」ではなく、**概要＋証拠（代表実績）＋次アクション導線（Projects/Contact）**に絞る。
- 詳細（ケーススタディ、設計の深掘り、回路図、GitHub等）は **/projects や個別ページ**へ分離する。

## 1. 現状の /profile 構成（2025-12-30時点の観測）

/profile の主要要素は概ね以下（※実ページ確認ベース）:
- 英語肩書き: “IoT hardware and software engineer”
- ナビ: Profile / Projects / Contact
- 目次
- Hero（プロフィール画像＋短いキャッチ）  
  - 「聞いて、整理して、作る。要件とプロダクトをつなぐエンジニアです。」
- セクション: 自己紹介 / 実績・経歴（折りたたみ詳細あり）/ スキル・強み（折りたたみ詳細あり）/ 主なプロジェクト（2件＋一覧導線）/ ODM供給・業務委託事例 / 連絡先

この骨格は良いです。校正の焦点は **「初見の数秒で刺さる情報の出し方」「読み飛ばし前提の構造」「非エンジニアに通じる言葉」**です。

---

## 2. 学術・実証知見（デザイン制約として使う）

このガイドは、以下の知見を“制約条件”として扱います（要点だけ記載）。

- Web閲覧では多くの人が**精読せずスキャン**する（新規ページはスキャンが主）  
- 平均的な訪問でユーザーが読むのは、ページ内テキストの**最大28%程度、現実的には20%程度**という報告がある  
- 視線はしばしば **F字型**のパターンで動く（上部→左側を優先して拾う）
- 採用文脈では「初期の目の動き（初見判断）」が短いというレポートがあり、**最初の画面で要点が伝わる設計**が重要

→ したがって /profile は、
1) ファーストビュー（最初の画面）で要点が分かる  
2) スキャンで拾える構造（見出し・箇条書き・短文・余白）  
3) 詳細は折りたたみ／別ページへ分離（Progressive Disclosure）  
を満たす必要があります。

## 3. Rubric（採点基準：0/1/2点）

採点は「改善の優先順位付け」のために使います。**Aは最重要（落ちると全体が崩れる）**。

### A. 7秒ファーストビュー（最重要）
- **2点**：ファーストビューだけで次が分かる  
  - ①誰（職能） ②強み3つ ③代表実績1つ（証拠） ④次に押す導線（Projects/Contact）
- **1点**：2〜3項目は分かるが、どれかが曖昧／導線が弱い
- **0点**：抽象的な文章が長い／次に何を見ればよいか不明

### B. スキャン前提の可読性（読まれない前提で設計できているか）
- **2点**：見出し・箇条書き・強調で要点が拾える。長文段落がない
- **1点**：一部拾えるが段落が長い／見出しが弱い
- **0点**：段落中心で重要語が埋もれる

### C. 視線誘導（Fパターン対応）
- **2点**：重要語が各行の前半（左側）に置かれ、情報の塊が規則的（カード/見出し/余白）
- **1点**：重要語はあるが配置が散る
- **0点**：要点の位置が不規則、目が迷う

### D. プロジェクト提示（価値→役割→成果）
- **2点**：代表2–4件をカード化し、各カードに「何/誰のため/担当/成果（数値or事実）」がある
- **1点**：カードはあるが成果が抽象、または担当が不明
- **0点**：羅列で差分が分からない

### E. 非エンジニアにも通じる平易さ
- **2点**：略語に補足。専門語は必要最小限。成果は事実ベースで説明
- **1点**：一部で専門語が続く
- **0点**：略語・専門語が多く理解不能

**合格ライン（Done）**：A=2 かつ、B〜E合計が **7点以上 / 10点**

## 4. 校正チェックリスト（エージェントがリポジトリ上で直接レビューする手順）

### 4.1 リポジトリ内の編集対象を特定する
- [ ] /profile のソースファイル（Markdown or テンプレ）を特定する  
  - 目印文字列で検索: `聞いて、整理して、作る` / `IoT hardware and software engineer` / `ODM供給` など
- [ ] 目次（ToC）生成がテーマ側か、ページ側かを確認する（自動ToC or 手動ToC）
- [ ] 折りたたみ（details/summary）が Markdown の拡張か shortcode かを確認する

### 4.2 ファーストビュー（最優先で直す）
- [ ] 「職能ラベル（何者か）」が明確（日本語も併記する場合は短く）
- [ ] 価値提案が**1文で言い切り**、抽象語だけになっていない
- [ ] 強みが **3つの短い箇条書き**（各行は短く、重要語を先頭へ）
- [ ] 代表実績が **1つだけ**ファーストビューで見える（例：ODM供給、雑誌掲載、展示運用など“事実”）
- [ ] CTA（導線）は **2つだけ**（例：Projects / Contact）  
  - 目標：迷わず次へ進める（選択肢の増やしすぎを避ける）

### 4.3 分量と構造（20–28%しか読まれない前提）
- [ ] 各セクション冒頭に **1行サマリ**（詳細は折りたたみ or 別ページ）
- [ ] 段落が3行以上続かない（続くなら箇条書きに分解）
- [ ] 見出しだけ追っても内容が分かる（見出しが具体）
- [ ] 目次がファーストビューを圧迫していない（必要なら折りたたみ or 下へ）
- [ ] 画像のキャプションが「何の証拠か」を補助している（飾りだけになっていない）

### 4.4 視線誘導（Fパターン）
- [ ] 重要キーワード（役割・成果・数字）が**左側/前半**に置かれている
- [ ] セクション内の情報の並びが規則的（同じ型の繰り返し）
- [ ] 強調（太字/色）は乱用せず、要点に限定している
- [ ] 余白が足りず詰め込みに見える箇所がない

### 4.5 プロジェクト（主なプロジェクト＋一覧への誘導）
各プロジェクトカード/抜粋に対し、以下が揃っているか:
- [ ] **何を作ったか**（名詞で言い切り）
- [ ] **誰の課題か**（対象ユーザー/利用シーン）
- [ ] **あなたの担当**（回路/FW/筐体/UI/運用など）
- [ ] **成果**（数値が理想。なければ事実：展示運用、量産、納品、掲載…）
- [ ] 次の導線（個別ページ/GitHub/動画等）を**1つに絞る**（迷わせない）

※現状は主なプロジェクトが2件なので、**「2件＋一覧」**は良い方針。  
ただしカード内の“成果の型”を揃えると、非エンジニアにも差分が伝わりやすい。

### 4.6 実績/経歴（折りたたみの使い方）
- [ ] 折りたたみの外側に、**人事が拾う要点（雑誌掲載・ODM・受賞等）だけ**を並べる
- [ ] 折りたたみ内の年表は、1行が長すぎない（媒体名やリンクは簡潔）
- [ ] “予定”が多い場合は、現時点の確定実績と分離する（確定/予定の混在を避ける）

### 4.7 スキル/強み（羅列を“証拠つき”に変える）
- [ ] スキルは「カテゴリ→代表ツール→根拠（どの実績で使ったか）」の順
- [ ] 年数表記がある場合、**根拠（代表成果）**を併記する（盛りに見えない）
- [ ] 略語（PSRR/LDO等）は括弧で一言補足（人事向け）
- [ ] 重要スキル上位5つが先に見える（残りは折りたたみ）

### 4.8 ODM供給・業務委託（信頼の“核”）
- [ ] 各案件に「担当」「成果」が明示されている（現状は良い）
- [ ] “匿名案件”は、読者が不安にならない範囲で **成果の具体性**を担保する（例：一次試作納品、帯域仕様達成など）
- [ ] NDA表現は簡潔で、面談で説明可能な範囲が分かる

### 4.9 連絡先・導線（機会損失を防ぐ）
- [ ] Contact への導線が上部（ナビ or CTA）にある
- [ ] Email/GitHub/Twitter が迷わず見つかる
- [ ] 外部リンクは新規タブ等、テーマ方針に合わせて統一

### 4.10 ビジュアル中心サイトとしての実装（最低限）
- [ ] 画像に `alt` があり、内容が伝わる
- [ ] 画像が重すぎない（圧縮・サイズ最適化・遅延読み込み）
- [ ] モバイル幅でもファーストビューが成立（要点がスクロール前に入る）

## 5. エージェントのレビュー出力フォーマット（PR/Issueコメント用）

以下のテンプレで、リポジトリ上に直接コメントしてください。

### 5.1 採点（Rubric）
- A（7秒ファーストビュー）: _/2  
- B（スキャン可読性）: _/2  
- C（視線誘導）: _/2  
- D（プロジェクト提示）: _/2  
- E（平易さ）: _/2  
- 合計: _/10（合格ライン：A=2 & 合計7以上）

### 5.2 指摘（優先度順）
- **Must（今すぐ直す）**:  
  - 例：ファーストビューに「強み3点＋代表実績1点＋CTA」を追加、目次の圧迫を解消、等
- **Should（次に直す）**:  
  - 例：プロジェクトカードの型統一、スキルの根拠付け、略語補足
- **Could（余裕があれば）**:  
  - 例：画像最適化、alt改善、微調整

### 5.3 具体的な提案（差分が作れる粒度）
- 修正対象ファイル: `content/...` / `layouts/...` など（実際に見つけたパスを書く）
- 提案する文言（Before/After）
- レイアウト変更（例：Heroに箇条書き追加、ToCを折りたたみ、カード化）
- 影響範囲（他ページ・CSS・テーマへの影響）
